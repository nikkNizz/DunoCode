DunoCode : a fantastic scripting language...

Characteristics
DunoCode is a scripting language based on simple sequences of unique directives
(Duno = Direttiva UNO), mainly adapted to Linux systems.
In DunoCode everything is an internal function with any parameters, including delimiters
blocks and variable assignment!

The interpreter is a simple program, written in any language, and is part
integral to the scripting process, as it provides the input graphical elements
and output and must always be launched in order to execute the code.

DunoCode is mainly aimed at 'on the fly' execution of scripts, but also:
  when executing files;
  to the creation of scripts with a partial graphical interface;
  to those who want to perform operations with a unique, schematic and simple syntax.
  at those who don't like the cryptic commands typical of bash (awk, sed, expr etc...).

Hello World and basic syntax

print >Hello World
#> comment
index = indexOf >Hello World ; W ;
print >\index\
end>

The lines of code are composed of the scheme: 
variable =  directive > parameters
Where directive is a predefined internal function; the > symbol separates it
by parameters that are separated from each other by semicolons(;). Any space
preceding the semicolon and the following one are not considered in the parameters.
The carriage return defines the end of the line and the 'end' directive closes the script.
Comments can be written after the '#' function or on any line that does not contain the symbol ( > ).
Comments can also be written on the same line of a directive, as last parameter. If preceded 
by a double slash (//), these parameters will be highlighted.

An empty directive is equal to  directive. 

When a variable is used in parameters it must be placed between two backslashes.
Concatenation between strings and variables does not require symbols.

Directives are not case sensitive (while parameters and variables are) and can be written with
multiple spaces.

Summary:
-the general syntax is:
      function without parameters >
  or: function > param1;param2;...
  or:  variable = function > param1 ; param2 ; ...
-the end of the line is defined by the carriage return
-the end of script is defined by the 'end' directive
-directives are not case sensitive
-parameters and variables are case sensitive
-comments can be written on any line where the symbol > is not present
 or after the comment directive '#', or as the last parameter
-the concatenation of data and variables does not require symbols
-variables are recalled by placing them between backslashes.

Errors
Errors are handled automatically.
If you receive error messages from the operating system, close DUNO.
A possible error in executing a file can be given by the ansi encoding
of the file. Save the files to be executed in UTF-8.

In case of missing parameters, the directive will still be executed if
can use blank or zero default values. Otherwise the script is stopped and 
the error reported.

Syntax errors will be reported.

Path not found. The script will run anyway.

Out-of-range items in a list. However, a value (!error) will be returned and
the script will run anyway.

Stack overflow for loop too long or infinite. The loop does not cause overflow.

When no errors are reported, but the code does not give expected results, try inserting
 directives in your code.
Also try highlighting the code and checking the script status from the interpreter menu.

BackScript or backward script is a convenient feature peculiar of DUNO.
It allows you to pass a file to be processed to a script... from the file itself.
We have a document (harvest.du) structured as follows:

Name  quintals
Paul: 100
Louis: 88
Mario: 70
Anna : 74

You want to find the total and average of the harvest without having to open every time
the calculator.
A script will be created only once, placed in a folder and forgotten.
In the data file it will be sufficient to insert the in marker exactly at the beginning
uppercase containing the script path:
<DUNOCODE_PROCESS>/Path/to/myScript</DUNOCODE_PROCESS>.
The path can be absolute or relative to the home (user in Windows).
The path /home/MARIA/Desktop/calculation is equivalent to /Desktop/calculation and related paths
have precedence.
By assigning the extension chosen for DunoCode scripts (typically .du), to the
data, it will be executed upon double clicking, redirecting to the script.
The document can still be opened by a text editor.
When the interpreter reads the header it will load /Path/to/myScript, appending
the following lines at the top of the same:

dataPath = mem>/path/to/harvest.du
dataText = fileRead>\dataPath\
head = mem><DUNOCODE_PROCESS>/Path/to/myScript</DUNOCODE_PROCESS>
dataText= replFirst>\dataText\;\head\;

In the script, use the dataText variable (and possibly dataPath and head
if you wanted to rewrite the source file.)
This approach also allows you to move the data file anywhere, being
it is enough that the script remains in its own folder.
If problems are encountered with paths or file names, the interpreter
will try to run the data source file (as the first file passed),
therefore it will not give the desired result.
The script we wrote could therefore be like this:

list = Listfrom>\dataText\;:|\n.\
tot = mem>0
num = mem>0
item = nextItem>\list\;1
  Iffloat>\item\
    tot = calc>\tot\ + \item\
    ++>\num\
  Endif>
next>
Notes>Total harvested \tot\\n.\Average x worker [--decimal>[--calc>\tot\ / \num\--]; 2--]
end>

When we launch the harvest.du file, the script will actually be executed,
supplemented by automatic lines.

External code.
DunoCode allows you to automatically copy lines of code from files, via
the pre-directive includes.

 <include>/path/to/file</include>
 
Wherever this line is present it will be replaced by the code present in the specified file.
To point directly to the Dunocode folder created by the program, use the colon at
start of path ( : corresponds to /home/$user/Dunocode ).

 <include>:/lib/file</include> 
 
The path must not contain variables.
The included code, of course, should not end with End>
If there is no file in the path or if the tags are written incorrectly (e.g. <i nclude>),
an error will be given. Be careful not to insert unnecessary spaces before or after the
path and keep the file name case-sensitive.
